# Финальный Глобальный Аудит Проекта VPN-manager
*Время проведения: 2026-04-08*

В ходе глубокого статического и архитектурного анализа кодовой базы (после интеграции последних фиксов из `dev`) были выявлены скрытые системные уязвимости. Основной упор делался на архитектуру синхронизации, работу с памятью (утечки сессий) и транзакционную модель базы данных.

Текущий код содержит **4 критических проблемы**, которые могут привести к неконтролируемым падениям или потере пользовательских данных. Ниже приведено детальное описание для будущих исправлений.

---

### 1. Критическая утечка памяти (Memory / Session Leak) в меню пользователя
* **Где:** `app/bot/handlers/user/subscriptions.py` (функция `show_subscription_status`).
* **Проблема:** Для показа актуального остатка трафика пользователя скрипт циклично открывает сетевые сессии `xui_client`. Очистка и закрытие этих сессий (`await client_obj.close()`) расположена в самом конце функции, **но без использования блока `finally`**.
* **Чем грозит:** Если при вызове финального `await callback.message.edit_text(...)` Telegram API выбросит штатное исключение (например, `MessageNotModified` из-за двойного клика пользователя по кнопке), выполнение функции прервется. Блок очистки никогда не сработает, и открытые сокеты `aiohttp` навсегда "повиснут" в оперативной памяти (зомби-сессии). При активной аудитории это приведет к исчерпанию пула подключений, Out of Memory (OOM) и падению бота.
* **Как лечить:** Обернуть блок изменения сообщения и закрытия клиентов в `try...finally` (аналогично тому, как это было исправлено в `main.py` и `sync_service.py`).

### 2. Архитектурная потеря данных (Односторонняя синхронизация)
* **Где:** `app/services/sync_service.py` (метод `_sync_inbound_clients`) и сервисы обновления.
* **Проблема:** Фоновая 5-минутная синхронизация работает в режиме **Pull-Only** (строгое и безусловное доверие данным из XUI-панели). 
* **Чем грозит:** Если администратор через бота меняет лимит трафика или срок подписки клиента, а XUI-панель в эту секунду отвечает таймаутом (или недоступна), локальная БД правильно сохраняет новые лимиты и ставит подключению флаг `sync_status = "error"`. 
Однако, когда запустится фоновый `SyncService`, он скачает старые данные из XUI-панели и **безвозвратно перезапишет ими локальную БД**. Намерения администратора (увеличенный трафик) будут молча стерты.
* **Как лечить:** Реализовать Push-синхронизацию. `SyncService` перед пулом должен проверять БД на наличие записей со статусом `"error"` или `"pending"` и сначала принудительно отправлять локальные изменения в XUI, и только потом стягивать статистику.

### 3. Осиротевшие (мертвые) подключения при сетевых сбоях
* **Где:** `app/services/new_subscription_service.py` (метод `add_inbound_to_subscription`).
* **Проблема:** Это прямое следствие предыдущего бага. Если при создании нового подключения (выдаче подписки клиенту) панель отвечает ошибкой, подключение сохраняется в БД со статусом `error`.
* **Чем грозит:** Из-за отсутствия Push-механизма бот **никогда** больше не попытается отправить этого клиента в панель. Фоновая синхронизация просто будет писать в лог `[WARN] Клиент не найден в XUI` и игнорировать его. Подключение навсегда останется осиротевшим и не будет работать у клиента, пока администратор не пересоздаст его вручную.
* **Как лечить:** Аналогично п.2 — добавить механизм "досылки" (Push) несинхронизированных InboundConnection в панель при старте `_sync_cycle`.

### 4. Бомба замедленного действия: DetachedInstanceError в Middleware
* **Где:** `app/bot/middlewares/auth.py`
* **Проблема:** В Middleware создается локальная сессия БД для извлечения пользователя, после чего сессия **закрывается** (`async with async_session_factory() as session: ...`), а извлеченный объект `client` передается дальше в цепочку хендлеров (`data["client"] = client`).
* **Чем грозит:** Объект `client` попадает в хендлеры будучи "оторванным" от сессии. На текущий момент бот не падает только потому, что во всех хендлерах используются базовые скалярные поля (`client.id`, `client.name`) или клиент запрашивается из БД заново. Если в будущем разработчик решит использовать ленивую загрузку и напишет в любом хендлере `len(client.subscriptions)`, бот мгновенно упадет с критической ошибкой `sqlalchemy.orm.exc.DetachedInstanceError`.
* **Как лечить:** Внедрить паттерн `db_session` middleware. Сессия должна создаваться в middleware, передаваться в `data["session"]`, и закрываться **после** `await handler(event, data)`, чтобы хендлеры работали внутри активного контекста базы данных.
